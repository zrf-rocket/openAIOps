
将AIOps平台拆分为子系统，通常有两种视角：**技术功能视角**（侧重数据处理与分析）和**运维业务视角**（侧重解决具体运维场景）。

最经典且通用的技术架构划分，借鉴了Gartner等机构的参考模型，可以将AIOps平台拆解为以下**5大核心子系统**。

### 1. 数据采集与集成子系统 (Data Ingestion & Integration Layer)

这是整个平台的“五官和手脚”，负责将所有分散的运维数据统一接入，并执行初步清洗。

-   **多源适配器**：包含针对各类监控系统、云平台、数据库、中间件的专用连接器。
-   **数据标准化模块**：将不同来源的指标、日志、追踪数据格式统一，生成标准的“全观测性数据模型”。
-   **遥测数据接收器**：原生支持接收OpenTelemetry、Prometheus等格式的数据。
-   **实时流处理引擎**：负责处理高吞吐、低延迟的数据流，如Kafka Streams或Flink任务，完成初步过滤和富化。

### 2. 存储与元数据管理子系统 (Storage & Topology Layer)

这是平台的“记忆和地图”，负责海量数据的低成本存储，以及维护各组件间的动态关系。

-   **时序数据库**：专门存储CPU、内存等时序指标数据。
-   **日志搜索引擎**：用于海量日志的全文检索和快速索引。
-   **图数据库**：专门存储服务间的调用依赖、基础设施的拓扑关系，支持快速的多级依赖查询。
-   **CMDB集成缓存**：从外部CMDB同步并缓存配置项信息，避免实时查询影响配置管理数据库性能。

### 3. 智能分析引擎子系统 (Analytics Engine Layer)

这是AIOps的“大脑”，是技术含金量最高的部分。它由多个独立或协作的算法模型构成。

-   **时序异常检测引擎**：集成LSTM、Prophet等算法，负责指标的动态基线、突变和趋势预测。
-   **日志模式挖掘引擎**：利用NLP和聚类算法，从非结构化日志中自动发现高频模式和异常模式。
-   **根因分析引擎**：结合时间序列相关性分析和拓扑传播概率，计算故障根源。
-   **事件关联与聚合引擎**：负责去重、降噪，并将相关告警聚合成一个智能事件。
-   **预测模型引擎**：管理容量预测、故障预测等各类时序预测任务。

### 4. 编排与自动化执行子系统 (Orchestration & Automation Layer)

这是平台的“手和脚”，负责将分析结果转化为实际行动。

-   **工作流引擎**：支持可视化的运维剧本设计，定义“检测-诊断-修复-验证”的步骤。
-   **任务执行器**：连接Ansible、SaltStack等工具，执行具体命令，如重启、清理日志。
-   **流程集成网关**：与ServiceNow、Jira等工单系统、以及钉钉、Slack等通讯工具集成。

### 5. 可视化与交互子系统 (Visualization & Interaction Layer)

这是平台的“脸”，也是运维人员最直接接触的部分。

-   **服务拓扑可视化**：动态展示服务地图，通常采用红绿灯或动画表示健康状态。
-   **统一告警工作台**：经过关联和聚合后的唯一告警入口，支持多维度筛选和屏蔽。
-   **智能对话/搜索模块**：新一代集成LLM的界面，允许用户通过自然语言提问。
-   **自定义仪表盘与报表**：为不同角色（如SRE、DBA）展示核心指标和SLO报表。

### 补充说明：按运维业务领域划分

除上述技术视角外，从业务功能角度，也可将AIOps拆分为几个独立产品：

-   **智能告警平台**：只负责告警的接入、抑制、去重和分派。
-   **根因分析系统**：专注于快速定位故障根源。
-   **容量预测系统**：侧重资源趋势预测和成本优化。
-   **日志智能分析系统**：用于分析海量日志，发现异常和潜在安全威胁。
-   **自动化运维平台（底层的执行能力）**：作为自动化底座，被AIOps平台调用。

## 一、数据采集与集成子系统

### 1.1 功能设计

数据采集与集成子系统是AIOps平台的“五官和手脚”，负责从各类IT基础设施、应用系统、中间件中采集原始运维数据，并进行初步的清洗、标准化和传输。

#### 1.1.1 核心功能模块

**（1）多源数据适配器**

| 数据源类型 | 采集方式 | 典型工具/协议 |
|-----------|---------|--------------|
| 主机/服务器 | Agent/Agentless | SSH、WinRM、SNMP、IPMI |
| 容器/K8s | API采集 | K8s API、cAdvisor、Kubelet |
| 数据库 | 日志解析/JDBC | Binlog、Redo Log、JDBC |
| 中间件 | 插件/Exporter | Prometheus Exporter、JMX |
| 云服务 | API集成 | AWS CloudWatch、Azure Monitor |
| 网络设备 | 协议采集 | SNMP、NetFlow、sFlow |
| 应用日志 | 日志采集 | Filebeat、Fluentd、Logstash |
| 链路追踪 | SDK/Agent | Jaeger、Zipkin、OpenTelemetry |
| 告警事件 | Webhook集成 | REST API、Kafka |

**（2）变更数据捕获（CDC）引擎**

CDC是实现实时数据采集的核心技术，其设计需满足以下要求：

- **零性能损耗**：采用“旁路监听”机制，直接读取数据库底层日志（如MySQL Binlog、Oracle Redo Log、PostgreSQL WAL），避免对源端业务产生性能影响
- **事务一致性保障**：引入全局序列号（GSN）或LSN对齐机制，确保跨库、跨表的事务原子性，防止脏读或数据错序
- **断点续传**：记录消费位点，故障恢复后自动从断点继续拉取，保证数据不丢不重

**（3）实时流处理引擎**

负责对采集到的数据进行实时处理：

- **数据过滤**：根据规则过滤无关或冗余数据
- **数据富化**：补充上下文信息（如服务名、环境标签、地理位置）
- **格式标准化**：将不同来源的数据转换为统一格式
- **流量整形**：支持背压机制，当下游处理能力不足时自动限流

**（4）采集任务管理**

- 任务生命周期管理（创建、启停、删除、更新）
- 采集频率/粒度配置（秒级、分钟级、小时级）
- 采集范围选择（全量、增量、指定对象）
- 采集状态监控与告警

### 1.3 关键设计要点

**1.3.1 零侵入采集设计**

采集器必须对源端业务系统做到“零感知”：
- 采用旁路日志解析，而非轮询查询
- Agent资源限制（CPU/内存上限、磁盘限制）
- 支持采集降级（源端压力高时自动降低采集频率）

**1.3.2 端到端数据质量保障**

- **数据校验**：支持源端与目标端的数据自动比对
- **数据去重**：基于事件ID实现幂等写入
- **数据延迟监控**：实时采集端到端延迟，超过阈值告警

**1.3.3 弹性伸缩能力**

- 采集任务可根据数据量动态分片
- 流处理作业支持自动并行度调整
- 消息队列支持分区动态扩展

## 二、存储与元数据管理子系统

### 2.1 功能设计

存储与元数据管理子系统是AIOps平台的“记忆和地图”，负责海量运维数据的持久化存储，以及IT组件间动态关系的维护。

#### 2.1.1 核心功能模块

**（1）时序数据存储**

用于存储各类性能指标数据（CPU、内存、QPS、延迟等）：

- 高吞吐写入：支持百万级TPS的指标写入
- 高效查询：支持时间范围查询、聚合查询、降采样查询
- 数据生命周期管理：自动归档、过期删除、冷热分层
- 高压缩率：采用专用压缩算法（如Gorilla压缩）节省存储空间

**（2）日志存储与检索引擎**

用于存储和检索海量日志数据：

- 全文检索：支持倒排索引，毫秒级日志检索
- 日志聚类：自动识别相似日志模式
- 实时索引：日志写入即可查询
- 多租户隔离：支持按项目/环境划分日志索引

**（3）链路数据存储**

用于存储分布式调用链数据：

- Trace存储：存储完整的调用链拓扑
- Span索引：支持按TraceID、服务名、操作名等维度查询
- 延迟分析：存储各Span的耗时信息

**（4）元数据管理（CMDB）**

配置管理数据库，存储IT资产的配置信息和关系：

| 元数据类型 | 内容示例 | 数据来源 |
|-----------|---------|---------|
| 资源元数据 | 服务器、网络设备、存储设备 | 自动发现 + 手动录入 |
| 应用元数据 | 应用服务、微服务、API | APM采集 + K8s API |
| 中间件元数据 | 数据库、缓存、消息队列 | Exporter采集 + CMDB同步 |
| 关系元数据 | 服务依赖、资源归属 | 拓扑自动发现 |
| 运维元数据 | 告警规则、监控阈值、SLO | 配置管理 |
| 组织元数据 | 项目归属、团队、负责人 | 企业目录同步 |

**（5）拓扑关系管理**

动态维护IT组件间的依赖关系：

- 服务拓扑：自动发现微服务间的调用关系
- 基础设施拓扑：服务器→集群→机房→数据中心层级关系
- 应用-资源映射：应用与所依赖基础设施的关联
- 变更追踪：拓扑变更历史记录与审计

**（6）数据生命周期管理（DLM）**

- 数据分级：热数据（近期）、温数据（近月）、冷数据（归档）
- 存储分层：SSD（热）→ HDD（温）→ 对象存储（冷）
- 自动清理：基于TTL策略自动删除过期数据
- 数据脱敏：敏感数据自动识别与脱敏处理

#### 2.2.3 关键设计要点

**2.2.3.1 时序数据高压缩**

采用Gorilla压缩算法：
- 时间戳压缩：Delta-of-Delta编码
- 数值压缩：XOR编码
- 典型压缩率：可将原始数据压缩至1/10~1/20

**2.2.3.2 日志索引优化**

- 使用倒排索引加速全文搜索
- 对日志字段进行分词和索引优化
- 支持索引生命周期管理（ILM），按时间滚动索引
- 热温冷索引分层：热索引（SSD，3天）→ 温索引（HDD，30天）→ 冷索引（对象存储，长期）

**2.2.3.3 图数据库拓扑存储**

在服务拓扑存储中，图数据库比关系型数据库有显著优势：
- 多跳查询性能：查询服务依赖链深度N的复杂度为O(N)，而非O(N*logN)
- 关系建模直观：节点=服务，边=调用关系
- 支持路径查询：最短路径、可达性分析、影响范围计算

**2.2.3.4 CMDB运维数据治理**

CMDB是AIOps的主数据源，需构建标准化治理体系：
- 数据资产化：构建运维数据资产地图
- 标准化治理：建立统一的数据模型和命名规范
- 数据运营化：持续提升数据质量，建立度量指标
- 自动化发现：通过多种采集方式自动发现并同步配置项

## 三、智能分析引擎子系统

### 3.1 功能设计

智能分析引擎是AIOps平台的“大脑”，是整个系统技术含金量最高的部分，负责从海量数据中提取洞察、检测异常、定位根因、预测趋势。

#### 3.1.1 核心功能模块

**（1）异常检测引擎**

异常检测是AIOps的基础能力，需超越传统静态阈值方式：

| 检测类型 | 算法 | 说明 |
|----------|------|------|
| **动态基线** | 时序分解（STL）、Prophet | 学习业务周期性，动态调整正常范围 |
| **突变检测** | 3-Sigma、IQR、MAD | 检测瞬时突增/突降 |
| **趋势检测** | 曼-肯德尔检验、线性回归 | 检测缓慢变化趋势 |
| **季节性检测** | 自相关分析、FFT | 识别周期性模式 |
| **多指标关联** | 皮尔逊相关系数、互信息 | 多指标联合异常判定 |
| **孤立点检测** | Isolation Forest、LOF | 高维空间异常识别 |

**（2）日志智能分析引擎**

对海量非结构化日志进行自动化分析：

- **日志模式聚类**：使用聚类算法（如LDA、DBSCAN）自动发现日志模式，将海量日志归为若干类别
- **异常日志识别**：从日志模式中识别异常模式，如新增错误类型、频率突变
- **日志实体提取**：使用命名实体识别（NER）提取关键实体（IP、用户ID、订单号等）
- **日志摘要生成**：自动生成故障期间的关键日志摘要
- **日志与指标关联**：将日志事件与性能指标时间轴对齐，辅助根因分析

**（3）根因分析引擎（RCA）**

这是AIOps最具挑战性也最有价值的功能：

- **因果推断**：基于时间序列因果分析（格兰杰因果、传递熵）推断事件因果关系
- **拓扑传播分析**：结合服务依赖拓扑，分析故障沿调用链的传播路径
- **变更关联**：自动关联配置变更、发布记录，判断是否为故障诱因
- **多源证据融合**：融合指标异常、日志错误、链路延迟等多源证据
- **置信度评估**：为根因结论提供置信度分数
- **知识图谱推理**：基于运维知识图谱进行符号推理

**（4）预测引擎**

实现从“被动救火”到“主动防火”的转变：

| 预测类型 | 算法 | 应用场景 |
|----------|------|---------|
| **容量预测** | 时间序列预测（ARIMA、Prophet、LSTM） | 预测CPU、内存、磁盘使用趋势，提前扩容 |
| **故障预测** | 生存分析、异常趋势外推 | 预测磁盘故障、内存泄漏、服务过载 |
| **成本预测** | 回归模型 | 预测云资源消耗趋势，辅助预算 |
| **调用量预测** | 时序预测 | 预测业务高峰期，提前准备资源 |

**（5）事件关联与聚合引擎**

解决“告警风暴”问题：

- **告警去重**：基于告警指纹去重，相同原因的多条告警只保留一条
- **告警压缩**：将短时间内的连续告警聚合为一个事件
- **告警关联**：将相关告警（同一根因、同一服务、同一时间段）聚合成事件集
- **告警抑制**：已知故障期间，抑制衍生告警
- **告警优先级计算**：基于影响范围、紧急程度智能排序

**（6）多智能体协作引擎**

利用多智能体架构实现复杂运维任务的协同处理：

| 智能体类型 | 职责 |
|-----------|------|
| **任务规划智能体** | 接收告警/用户问题，拆解为分析计划，调度其他智能体 |
| **指标分析智能体** | 查询时序数据，执行异常检测，输出异常时间点和异常指标 |
| **日志分析智能体** | 检索并分析日志，提取错误堆栈和关键事件 |
| **链路分析智能体** | 分析调用链数据，识别慢请求和故障传播路径 |
| **拓扑感知智能体** | 查询服务依赖关系，分析故障影响范围 |
| **决策仲裁智能体** | 整合各智能体证据，做出根因判断和处置决策 |
| **报告生成智能体** | 生成结构化分析报告，包含结论、证据链和建议 |

### 3.3 关键设计要点

**3.3.1 动态基线算法**

传统静态阈值已无法适应现代动态负载：

```
静态阈值问题：
- 夜间业务低谷，CPU 30%正常，但静态阈值60%无法触发告警
- 大促期间，CPU 70%正常，但静态阈值60%产生大量误报

动态基线方案：
- 以天为单位学习周期性：工作日vs周末、白天vs夜间、月初vs月末
- 以周为单位学习长期趋势：业务增长带来的基线漂移
- 输出：正常范围的上限和下限（如P5-P95分位数带）
- 异常判定：实际值持续超出基线范围
```

**3.3.2 根因分析置信度机制**

为防止“幻觉”导致错误结论，需建立置信度阈值机制：

```
置信度计算维度：
1. 证据充分度：支撑结论的证据条数（指标异常数+日志错误数+变更关联度）
2. 证据一致性：各证据是否指向同一根因
3. 时序合理性：因在前，果在后，时间差合理
4. 拓扑合理性：沿调用链向上游传播，符合依赖关系

判定规则：
- 置信度 ≥ 0.8：可直接输出根因结论
- 0.5 ≤ 置信度 < 0.8：输出候选根因，标注“需人工确认”
- 置信度 < 0.5：仅输出异常事实，不给出根因判断
```

**3.3.3 知识库建设（RAG）**

为提升大模型分析准确性，需建立运维知识库：

| 知识类型 | 内容 | 用途 |
|----------|------|------|
| 系统架构文档 | 服务拓扑、依赖关系、技术栈 | 理解系统结构 |
| 历史故障案例 | 故障现象、根因、解决方案、修复时间 | 案例匹配、方案推荐 |
| Runbook手册 | 标准操作流程、应急预案 | 故障处理指引 |
| 变更记录 | 代码变更、配置变更、发布历史 | 变更关联分析 |
| SLO/SLI定义 | 服务等级目标与指标 | 影响范围评估 |

## 四、编排与自动化执行子系统

### 4.1 功能设计

编排与自动化执行子系统是AIOps平台的“手和脚”，负责将智能分析的结果转化为实际的操作动作，实现“检测-诊断-修复-验证”的自动化闭环。

#### 4.1.1 核心功能模块

**（1）工作流编排引擎**

支持可视化的运维流程设计：

- **流程设计器**：拖拽式流程编排，支持顺序、分支、并行、循环等控制结构
- **原子操作库**：预置常用运维操作原子（重启、清理、扩容、切流等）
- **条件判断**：支持基于指标、时间、状态的流程分支
- **超时控制**：流程执行超时自动中断
- **重试策略**：可配置的重试次数和退避策略
- **并行执行**：支持多任务并行执行，提升效率

**（2）任务执行器**

负责执行具体的运维操作：

| 执行对象 | 执行方式 | 典型操作 |
|----------|---------|---------|
| 服务器 | SSH/Ansible/SaltStack | 服务重启、日志清理、脚本执行 |
| Kubernetes | K8s API | Pod重启、滚动更新、扩缩容 |
| 数据库 | JDBC/CLI | 会话终止、参数调整、备份恢复 |
| 云资源 | 云厂商API | 实例扩容、磁盘扩展、快照创建 |
| 网络设备 | NETCONF/SSH | 路由调整、链路切换 |
| 中间件 | Admin API | 缓存清理、队列重置、连接池调整 |

**（3）自动化修复场景库**

预置常见故障的自动化修复方案：

| 故障场景 | 检测方式 | 修复动作 | 验证方式 |
|----------|---------|---------|---------|
| 服务进程崩溃 | process_up=0 | systemctl restart | process_up=1 |
| 磁盘空间不足 | disk_free<10% | 清理过期日志 | disk_free>20% |
| CPU持续高负载 | cpu_usage>80%持续10分钟 | 慢查询分析+限流 | cpu_usage<60% |
| 内存泄漏 | memory_usage持续增长 | 服务重启 | memory_usage下降 |
| 数据库连接池满 | active_connections=99% | 终止长连接+调整max_connections | 连接池释放 |
| 网络丢包严重 | packet_loss>5% | 切换备链路 | packet_loss<1% |

**（4）变更管理**

- **变更请求（CR）**：自动修复前生成变更请求，需人工审批或自动审批
- **回滚机制**：每次变更前备份状态，异常时自动回滚
- **审计追踪**：记录每次自动化操作的完整链路（who/what/when/why/result）
- **变更窗口**：支持配置变更执行的时间窗口，避免业务高峰期操作

**（5）闭环验证**

自动化操作执行后的效果验证：

```
闭环验证流程：
1. 执行修复动作
2. 等待指定时间（如30秒，等待服务稳定）
3. 查询关键指标验证恢复情况
4. 若指标恢复正常 → 记录成功，结束流程
5. 若指标未恢复 → 
   a. 尝试回滚操作
   b. 回滚后再次验证
   c. 回滚失败或仍然异常 → 升级为人工处理
```

**（6）安全与权限控制**

自动化执行涉及高风险操作，需有严密的安全机制：

- **最小权限原则**：执行器使用临时凭证，仅授予本次操作所需的最小权限
- **操作白名单**：仅允许执行白名单内的操作类型和目标
- **熔断机制**：单位时间内同一目标的操作次数超过阈值时自动熔断
- **审批流程**：高危操作（如切流量、删除数据）需人工审批
- **操作限速**：对同一目标的操作施加时间限制，避免频繁操作

### 4.3 关键设计要点

**4.3.1 幂等性设计**

自动化操作必须是幂等的，同一操作执行多次与执行一次结果相同：

- 采用幂等API或命令（如K8s apply而非create）
- 操作前检查状态，避免重复操作（如服务已运行时不再重启）
- 使用分布式锁防止并发执行

**4.3.2 回滚机制**

自动修复必须配备回滚能力：

```
回滚策略：
1. 状态快照：执行变更前记录当前状态
2. 操作原子性：变更操作要么全部成功，要么全部失败
3. 自动回滚触发条件：
   - 验证检查失败（指标未恢复）
   - 执行超时
   - 出现新的严重告警
4. 回滚验证：回滚后重新验证指标
```

**4.3.3 渐进式自动化**

从低风险到高风险逐步推进自动化：

| 等级 | 特点 | 示例 |
|------|------|------|
| L1-只读诊断 | 仅分析，不执行修复 | 根因分析，输出报告 |
| L2-建议模式 | 给出修复建议，人工确认后执行 | 索引优化建议，DBA审核 |
| L3-半自动 | 低风险操作自动执行，高危需审批 | 日志清理自动，切流需审批 |
| L4-全自动 | 全部操作自动执行（需充分验证） | 服务重启、Pod自动恢复 |
| L5-预测执行 | 故障发生前自动预防 | 预测磁盘满，自动清理 |

**4.3.4 审计与合规**

所有自动化操作必须可追溯：

```
审计日志字段：
- 触发源：告警/定时/API/人工
- 操作时间：开始、结束、耗时
- 操作目标：目标资源标识
- 操作内容：执行的命令/API
- 操作结果：成功/失败/回滚
- 验证结果：指标前后对比
- 操作人/审批人：系统账户或用户
```

### 5.1 功能设计

可视化与交互子系统是AIOps平台的“脸面”，是运维人员与AIOps能力交互的主要界面，负责将复杂的运维数据和分析结果以直观的方式呈现。

#### 5.1.1 核心功能模块

**（1）统一告警工作台**

经过事件关联和聚合后的唯一告警入口：

- **告警详情**：展示告警详情，包含指标趋势图、相关日志、影响分析、处理建议
- **告警认领**：支持运维人员认领告警，避免重复处理
- **告警屏蔽**：支持临时屏蔽已知告警（如维护窗口期间）
- **告警沉降**：支持将告警标注为误报或已知问题，供模型训练
- **告警聚合视图**：展示告警按服务、按类型、按时段的聚合统计

**（2）服务拓扑可视化**

动态展示服务依赖关系和健康状态：

- **拓扑地图**：以节点（服务）和边（调用关系）展示服务拓扑
- **健康着色**：节点颜色表示健康状态（绿/黄/红/灰）
- **故障传播动画**：展示故障沿调用链的传播路径
- **下钻分析**：点击节点查看该服务的详细指标和告警
- **时间轴回放**：支持回放历史时间点的拓扑状态，用于复盘

**（3）仪表盘与报表**

为不同角色提供定制化的数据视图：

| 角色 | 核心视图 | 用途 |

| SRE | 服务可用性、错误预算、MTTR趋势 | 服务质量监控 |
| DBA | 数据库性能大盘、慢查询排行 | 数据库运维 |
| 应用Owner | 应用QPS/延迟/错误率、依赖分析 | 应用性能监控 |
| 成本负责人 | 资源使用趋势、成本分析、优化建议 | 成本管理 |

### 5.3 关键设计要点

**5.3.1 实时数据流展示**

运维监控要求数据实时性，前端需支持：

- **WebSocket长连接**：实时推送新告警、指标更新
- **Server-Sent Events (SSE)**：用于单向实时推送（如指标图表更新）
- **增量更新**：仅传输变化的数据，减少带宽消耗
- **本地缓存**：缓存静态数据（如拓扑结构），减少重复请求

**5.3.2 大数据量渲染优化**

- **虚拟滚动**：处理告警列表等长列表时，仅渲染可视区域
- **数据采样**：时间跨度过大时，对指标数据进行采样展示
- **懒加载**：按需加载图表、模块
- **Web Worker**：将数据处理（如时序数据聚合）移至Worker线程，避免阻塞UI

**5.3.3 拓扑图高性能渲染**

服务拓扑图可能包含数百个节点，需优化渲染性能：

- **Canvas渲染**：大量节点时使用Canvas而非SVG
- **力导向算法优化**：使用Web Worker计算节点位置
- **节点聚合**：将同类型服务聚合成一个节点展示
- **分层展示**：按业务领域分层展示，逐层下钻

**5.3.4 响应式设计**

- 支持PC端（大屏、桌面）和移动端（手机、平板）自适应
- 移动端采用移动优先的响应式布局
- 针对移动端优化交互方式（手势、简化的视图）

## 七、总结与建设建议

### 7.1 子系统建设优先级

根据实际落地经验，建议按以下优先级建设各子系统：

| 优先级 | 子系统 | 核心价值 | 前置依赖 |
|--------|--------|---------|---------|
| P0（必须） | 数据采集与集成 | 没有数据，一切无从谈起 | 无 |
| P0（必须） | 存储与元数据管理 | 数据需要组织和持久化 | 数据采集 |
| P1（核心） | 可视化与交互 | 让数据可见、可操作 | 存储子系统 |
| P2（进阶） | 智能分析引擎 | 从自动化到智能化 | 存储子系统 |
| P3（高阶） | 编排与自动化 | 实现闭环自愈 | 前四项 |

### 7.2 技术选型建议

| 子系统 | 开源方案 | 商业方案 | 自研建议 |
|--------|---------|---------|---------|
| 数据采集 | Prometheus + Fluent Bit + OpenTelemetry | Datadog Agent | 集成开源，避免重复造轮子 |
| 时序存储 | VictoriaMetrics | InfluxDB Enterprise | 直接采用VictoriaMetrics |
| 日志存储 | OpenSearch | Splunk | 中小规模用Loki |
| 链路存储 | Jaeger + Cassandra | Dynatrace | 采用Jaeger |
| 智能分析 | 自研 + 开源模型 | Dynatrace Davis | 算法工程化投入大，建议购买或合作 |
| 自动化 | Ansible + K8s API | ServiceNow | 脚本级自研即可 |
| 可视化 | Grafana + 自研 | Dynatrace UI | Grafana可覆盖大部分场景 |

### 7.3 关键成功因素

1. **数据质量是根本**：AIOps的效果高度依赖数据质量，需投入精力治理元数据和数据一致性
2. **循序渐进**：不要试图一步到位实现全自动自愈，从“可观测→诊断→建议→半自动→全自动”逐步演进
3. **安全第一**：自动化修复必须有回滚、审计、熔断等安全机制
4. **人机协同**：AIOps的目标是赋能人而非替代人，将人从重复劳动中解放出来做更高价值的判断
5. **持续迭代**：通过反馈闭环不断优化模型和规则，形成“数据→洞察→行动→反馈→优化”的正循环
